## Coding Style & Naming Conventions

### General Rules
- **Indentation:** 2 spaces (no tabs)
- **Line length:** Maximum 100 characters
- **Quotes:** Use single quotes for strings (except JSON)
- **Semicolons:** Always use semicolons

### Naming Conventions

| Type | Convention | Example |
|------|-----------|---------|
| Files | kebab-case | `user-profile.ts` |
| Components | PascalCase | `UserProfile.tsx` |
| Functions | camelCase | `getUserData()` |
| Constants | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| Interfaces/Types | PascalCase with `I` prefix (optional) | `IUserData` or `UserData` |
| Private methods | camelCase with `_` prefix | `_internalMethod()` |

### Code Style Tools
- **ESLint:** Enforces code quality rules
- **Prettier:** Handles code formatting
- **TypeScript:** Provides static type checking

Run `npm run lint:fix` before committing to ensure code meets style guidelines.

## Testing Guidelines

### Testing Framework
- **Unit Tests:** Jest / Vitest
- **Integration Tests:** Testing Library
- **E2E Tests:** Playwright / Cypress (if applicable)

### Test File Naming
- Unit tests: `*.test.ts` or `*.spec.ts`
- Place test files next to the code they test or in `tests/` directory

### Coverage Requirements
- Aim for **minimum 80% code coverage**
- All new features must include tests
- Bug fixes should include regression tests

### Test Structure
```typescript
describe('ComponentName', () => {
  it('should perform expected behavior', () => {
    // Arrange
    const input = setupTestData();
    
    // Act
    const result = functionUnderTest(input);
    
    // Assert
    expect(result).toBe(expectedValue);
  });
});
```

### Running Tests
```bash
npm test                    # Run all tests once
npm run test:watch         # Watch mode for active development
npm run test:coverage      # Generate coverage report
```

## Commit & Pull Request Guidelines

### Commit Message Format
Follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Types:**
- `feat:` New feature
- `fix:` Bug fix
- `docs:` Documentation changes
- `style:` Code style changes (formatting, no logic change)
- `refactor:` Code refactoring
- `test:` Adding or updating tests
- `chore:` Maintenance tasks, dependencies

**Examples:**
```
feat(auth): add OAuth2 login support

fix(api): handle null response in user endpoint

docs: update installation instructions

test(utils): add tests for date formatting
```

### Pull Request Guidelines

**Before submitting a PR:**
1. âœ… Run all tests and ensure they pass

4. âœ… Add tests for new features

**PR Title:** Use the same format as commit messages

**PR Description should include:**
- **What:** Brief description of changes
- **Why:** Reason for the change
- **How:** Technical approach (if not obvious)
- **Testing:** How to test the changes
- **Screenshots:** For UI changes
- **Related Issues:** Link to issue(s) using `Fixes #123` or `Relates to #456`

## Security & Configuration Tips

### Environment Variables
- Never commit `.env` files
- Use `.env.example` as a template
- Document all required environment variables
- Use strong, unique values for production

### Sensitive Data
- Never hardcode API keys, passwords, or tokens
- Use environment variables or secure vaults
- Review code for accidental credential exposure

### Dependencies
- Regularly update dependencies: `npm audit`
- Review security advisories
- Use `npm audit fix` to auto-patch vulnerabilities

## Architecture Overview

### Design Patterns
- **Component-based architecture** for UI
- **Service layer** for business logic
- **Repository pattern** for data access
- **Dependency injection** for testability

### State Management
- Use appropriate state management solution (Context API, Redux, Zustand, etc.)
- Keep state minimal and normalized
- Avoid prop drilling with context or composition

### API Integration
- Centralize API calls in service modules
- Use consistent error handling
- Implement request/response interceptors for common logic
